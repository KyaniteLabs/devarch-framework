# Architecture Decision Records

Key design decisions and their rationale. Read this when onboarding or questioning why things are structured a certain way.

## ADR-001: Python as sole language

**Decision:** DevArch is pure Python. No JavaScript, no TypeScript, no shell scripts for core logic.

**Why:** The framework processes git history, runs analysis, and generates reports. All of these are data transformation tasks that Python handles well. Adding a second language would double the dependency surface with no functional benefit.

**Alternatives considered:** Node.js (rejected — no advantage for data processing), shell scripts (rejected — poor error handling, hard to test).

## ADR-002: SQLite as database engine

**Decision:** Use SQLite with FTS5 for commit storage and querying. No external database server.

**Why:** DevArch is a local-first CLI tool. Users run it on their machines against their repos. SQLite requires zero configuration, ships with Python, and handles the scale of even very large git repositories. FTS5 provides full-text search over commit messages without additional infrastructure.

**Alternatives considered:** PostgreSQL (rejected — requires server setup), DuckDB (considered for future — better analytical queries).

## ADR-003: Template-based HTML visualization

**Decision:** Visualizations are generated by hydrating HTML templates with data, not by a JavaScript framework.

**Why:** The output must be a single self-contained HTML file that works offline. No build step, no node_modules, no framework runtime. Chart.js is loaded from CDN for charts but the page works without it (graceful degradation).

**Alternatives considered:** React/Next.js dashboard (rejected — requires server), Plotly/Dash (rejected — heavy dependency).

## ADR-004: 9-stage pipeline with numbered folders

**Decision:** The pipeline is structured as 9 numbered stages (01-setup through 09-strategy), following ICM conventions.

**Why:** Numbered prefixes encode execution order in the filesystem. Alphabetical sorting displays them in correct sequence. Each stage has a CONTEXT.md with explicit Inputs/Process/Outputs contracts. This makes the pipeline self-documenting and inspectable at every stage boundary.

**Alternatives considered:** Single monolithic script (rejected — no inspectability), DAG-based pipeline (rejected — overkill for sequential flow).

## ADR-005: Sanitized product version

**Decision:** This repo (devarch-framework) is the product version. dev-archaeology is the lab version with real data. The product repo must never contain real project data.

**Why:** The framework is open source. Shipping real commit data, YouTube watch history, or session logs would be a privacy violation. All demo data is synthetically generated.

## ADR-006: 6 analysis vectors as markdown prompts

**Decision:** Analysis vectors are defined as markdown files in analysis-vectors/, not as Python code.

**Why:** The vectors are prompts for AI-assisted analysis. Markdown is the natural format for prompts. Keeping them as files means they can be inspected, edited, and versioned without touching code. Non-technical users can modify analysis behavior by editing markdown.

## ADR-007: ICM layer separation (2026-05-11)

**Decision:** Refactored CLAUDE.md, CONTEXT.md, and stage contracts to follow ICM 5-layer context hierarchy.

**Why:** The original CLAUDE.md was 192 lines mixing L0 (map), L1 (routing), and L3 (reference) content. Per ICM principle, each layer should load only what it needs. This refactoring reduced CLAUDE.md to pure L0 map, moved concepts to shared/concepts.md (L3), and added explicit L3/L4 separation to all stage contracts.

**Impact:** Agents now receive focused context per task instead of loading everything. Token budget reduced per stage. Workspace silos prevent cross-contamination between extraction, analysis, and reporting contexts.
